🧪 Testing Strategy

NextNotes Comprehensive Testing Guide

Overview

NextNotes implements a multi-layered testing strategy covering unit tests, integration tests, end-to-end tests, linting, and accessibility testing. This ensures code quality, reliability, and user experience.

✓ Status: Full testing suite configured with Vitest (unit), Playwright (e2e), ESLint (linting), and Axe (accessibility).

Testing Pyramid

We follow the testing pyramid principle: many unit tests, fewer integration tests, and even fewer e2e tests.

📦 Unit Tests (Many)
🔗 Integration Tests (Some)
🌐 E2E Tests (Few)

Developer Testing Workflow

During Development

  1. Write code in feature branch
  2. Run linter: bun run lint --fix to catch issues early
  3. Run tests: bun run test to verify functionality
  4. Check accessibility: bun run test:e2e tests/e2e/accessibility.spec.ts

Before Committing

  1. Run full test suite: bun run test
  2. Run all e2e tests: bun run test:e2e
  3. Run linter: bun run lint
  4. Fix any failures before committing

Writing Tests – Guidelines

Unit Tests (Vitest)

import { describe, it, expect } from "vitest";

describe("MyComponent", () => {
  it("should render with correct text", () => {
    // Arrange: Set up test data
    const props = { title: "Test" };

    // Act: Perform the action
    const result = renderComponent(props);

    // Assert: Verify the outcome
    expect(result).toContain("Test");
  });

  it("should handle errors gracefully", () => {
    expect(() => riskyFunction()).not.toThrow();
  });
});

E2E Tests (Playwright)

import { test, expect } from "@playwright/test";

test.describe("Authentication", () => {
  test("user can sign in", async ({ page }) => {
    // Navigate to page
    await page.goto("/authenticate");

    // Fill and submit form
    await page.fill("input[name='email']", "user@test.com");
    await page.fill("input[name='password']", "password123");
    await page.click("button:has-text('Sign In')");

    // Assert navigation
    await expect(page).toHaveURL(/\/dashboard/);
  });
});

Best Practices

✅ DO • Test behavior, not implementation
• Use descriptive test names
• Keep tests isolated and independent
• Use page objects for e2e
• Test error scenarios
• Mock external APIs
❌ DON'T • Test implementation details
• Create interdependent tests
• Use hardcoded wait times
• Test the testing framework
• Ignore flaky tests
• Make tests too complex

Common Test Scenarios

Testing Authenticated Routes (E2E)

async function signIn({ page }: { page: Page }) {
  const res = await page.request.post("/api/auth/sign-in/email", {
    data: { email: "user@test.com", password: "password123" }
  });
  expect(res.ok()).toBeTruthy();
}

test("authenticated user can access dashboard", async ({ page }) => {
  await signIn({ page });
  await page.goto("/dashboard");
  await expect(page).toHaveURL(/\/dashboard/);
});

Testing API Responses

test("API returns correct data", async ({ page }) => {
  const response = await page.request.get("/api/notes");
  expect(response.status()).toBe(200);
  const data = await response.json();
  expect(Array.isArray(data)).toBeTruthy();
});

Testing Accessibility

test("page has correct heading hierarchy", async ({ page }) => {
  await page.goto("/");
  const headings = await page.locator("h1, h2, h3").all();

  // Verify sequential hierarchy
  for (let i = 0; i < headings.length - 1; i++) {
    const current = await headings[i].evaluate(el => el.tagName);
    const next = await headings[i + 1].evaluate(el => el.tagName);
    const diff = Math.abs(parseInt(next[1]) - parseInt(current[1]));
    expect(diff).toBeLessThanOrEqual(1);
  }
});

Debugging Tests

Debug Vitest Tests

bun run test:ui

Opens interactive UI dashboard showing test results and filtering.

Debug Playwright Tests

npx playwright test --debug

Launches Playwright Inspector with step-by-step execution control.

Generate Playwright Report

npx playwright show-report

Displays detailed test report with videos, screenshots, and logs.

Common Debugging Techniques

  • Add await page.pause() to pause test execution
  • Use await page.screenshot({ path: "debug.png" }) to capture screenshots
  • Check playwright-report/ for HTML reports with videos
  • Use console.log() in tests (output visible in CLI)
  • Run single test: npx playwright test auth.spec.ts

CI/CD Integration

Tests should run automatically in your CI pipeline on every push and PR. Suggested workflow:

1. Lint: bun run lint
2. Unit Tests: bun run test
3. Build: bun run build
4. E2E Tests: bun run test:e2e
5. Coverage Report: bun run test:coverage

Fail CI if: Any lint errors, test failures, or coverage below targets.

Troubleshooting Tests

Flaky Tests (Intermittent Failures)

Problem: Tests pass sometimes, fail other times.

Solutions:

  • Replace timeouts with proper waitFor conditions
  • Use await page.waitForLoadState("networkidle")
  • Avoid hardcoded delays
  • Check for race conditions in async code

Port Already in Use (E2E Tests)

Problem: EADDRINUSE: address already in use

Solution: Kill existing process on port 3000:

lsof -ti:3000 | xargs kill -9

Playwright Tests Won't Run

Problem: Tests hang or don't start.

Solutions:

  • Ensure dev server is running: bun run dev
  • Check Playwright is installed: bunx playwright install
  • Clear Playwright cache: rm -rf ~/.cache/ms-playwright

Import Errors in Tests

Problem: Module not found errors in test files.

Solution: Verify path aliases in:

  • tsconfig.json (path mappings)
  • vitest.config.ts (alias configuration)
